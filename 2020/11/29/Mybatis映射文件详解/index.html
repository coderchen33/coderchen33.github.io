1.2 #{}和${}的区别

使用#{}

  • 使用 #{} 参数语法时，MyBatis 会创建 PreparedStatement 参数占位符，即在使用jdbc时的preparedStatement
  • sql语句中如果存在参数则会使用?作占位符，我们知道这种方式可以防止sql注入
  • 在组成sql语句的时候把参数默认为字符串带有引号，例，select * from table1 where id=#{id} 在调用这个语句时我们可以通过后台看到打印出的sql为：select * from table1 where id='2' 加入传的值为2

使用${}

  • 使用${}时的sql不会当做字符串处理，是什么就是什么，如上边的语句：select * from table1 where id=${id} 在调用这个语句时控制台打印的为：select * from table1 where id=2，2没有’’
  • 会发生sql注入的问题。

总结：我们最好是能用#{}则用它，因为它可以防止sql注入，且是预编译的，在需要原样输出时才使用${}

2.1resultType和resultMap

首先回顾下我们之前写的sql映射语句:

可以看到如果结果是实体类型，我们都是直接使用的resultType进行指定。为什么我们直接指定resultType就能将数据库中的列和实体类中的属性一一对应？原因在于：

  • 数据库列名和JavaBean中的属性名完全相同
  • MyBatis 会在幕后自动创建一个 ResultMap，再根据属性名来映射列到 JavaBean 的属性上。

如果我们的数据库列名和JavaBean的属性名不一致，如何解决?。

  • 使用别名的方式 (不推荐)
  • 使用resultMap进行显式指定。

2.3 自动映射

除开上面的一些复杂场景。在一些简单的场景下，MyBatis 可以为你自动映射查询结果。这里的自动映射也分两种。

  • 数据库列名和JavaBean属性名相同(不区分大小写的相同) MyBatis 会自动进行映射不需要处理。

  • 数据库列单词间用下划线分隔；而 Java 属性使用驼峰命名法约定。只需要将 mapUnderscoreToCamelCase 设置为 true。

2.4 结果集映射规律总结

  • 结果是基本类型，可以省略不写。Mybatis会进行自动推断。
  • 结果是一对一的JavaBean类型
    • 数据库列名和JavaBean属性名相同，直接使用resultType即可。
    • 数据库列名和JavaBean可以自动映射的，也是直接使用resultType即可。
    • 其他情况，使用万能的resultMap进行映射
  • 结果是一对多类型的使用resultMap进行映射
    • 结果不需要映射到集合中，使用<association>进行处理
    • 需要映射到集合中，使用collection进行处理，注意ofType指定的是映射到集合中的pojo类型。
